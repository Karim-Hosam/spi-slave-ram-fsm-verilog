## 3. What Is SPI?

**SPI (Serial Peripheral Interface)** is one of the most popular synchronous serial communication protocols. It enables full-duplex communication between a master device and one or more slave devices using four wires:

| Wire | Full Name | Direction (Master ↔ Slave) | Purpose |
|---|---|---|---|
| **MOSI** | Master-Out-Slave-In | Master → Slave | Serial data from master to slave |
| **MISO** | Master-In-Slave-Out | Slave → Master | Serial data from slave to master |
| **SCK** | Serial Clock | Master → Slave | Clock signal generated by master |
| **SS_n** | Slave Select (active-low) | Master → Slave | Selects the target slave |

### Key Characteristics

- **Full-duplex** communication (data flows both ways simultaneously).
- **High data rates** compared to I²C and UART.
- **No addressing** — slave selection is via dedicated SS_n lines.
- **Simple protocol** — no start/stop bits, no acknowledgement overhead.

> **Note:** In this project, the on-board 100 MHz oscillator on the Basys 3 serves as the system clock (`clk`).  There is no separate SCK wire because the SPI slave operates on the same clock domain as the rest of the design.

## 4. SPI Slave — How It Works

The `SPI_Slave` module is the core controller.  It:

1. **Receives serial data** bit-by-bit on MOSI and converts it to a 10-bit parallel word (`rx_data[9:0]`).
2. **Transmits data** from the RAM back to the master on MISO, converting 8-bit parallel data to serial output.
3. **Asserts `rx_valid`** when a complete 10-bit word has been assembled, signalling the RAM to accept it.
4. Uses an internal **4-bit up-counter** (`Up_Counter`) to track the current bit position during serial-to-parallel and parallel-to-serial conversion.
5. Maintains an internal flag **`rd_addr_Done`** to remember whether a read address has already been received, so the FSM can distinguish between READ_ADD and READ_DATA on subsequent read commands.

### SPI Command Sequences

#### Write Address (din[9:8] = 2'b00)
1. Master pulls SS_n LOW.
2. MOSI bit 0 = `0` → write operation detected at CHK_CMD.
3. Next 2 bits (`00`) form the command code.
4. Remaining 8 bits carry the write address.
5. After 10 bits: `rx_data` assembled, `rx_valid` asserted → RAM stores `din[7:0]` as `wr_addr`.
6. Master pulls SS_n HIGH to end communication.

#### Write Data (din[9:8] = 2'b01)
Same flow as above, but the command code is `01`.  RAM writes `din[7:0]` into `mem[wr_addr]`.

#### Read Address (din[9:8] = 2'b10)
1. Master pulls SS_n LOW.
2. MOSI bit 0 = `1` → read operation detected at CHK_CMD.
3. Command code `10` and 8-bit read address are shifted in.
4. RAM stores `din[7:0]` as `rd_addr`.  `rd_addr_Done` is set to 1.

#### Read Data (din[9:8] = 2'b11)
1. Master pulls SS_n LOW.
2. MOSI bit 0 = `1` and `rd_addr_Done == 1` → FSM enters READ_DATA.
3. Command code `11` + 8 dummy bits are shifted in (ignored by RAM).
4. RAM asserts `tx_valid` and places `mem[rd_addr]` on `dout`.
5. SPI slave shifts `tx_data` out serially on MISO, MSB first.
6. `rd_addr_Done` is cleared to 0.

## 5. Single-Port Synchronous RAM

The `Single_Port_Sync_RAM` module implements a **256 × 8-bit synchronous RAM** with the following interface:

| Port | Width | Direction | Description |
|---|---|---|---|
| `clk` | 1 | Input | System clock |
| `rst_n` | 1 | Input | Synchronous active-low reset |
| `din` | 10 | Input | Data input — `din[9:8]` = command, `din[7:0]` = address/data |
| `rx_valid` | 1 | Input | HIGH when `din` is valid |
| `dout` | 8 | Output | Data output (read result) |
| `tx_valid` | 1 | Output | HIGH when `dout` holds valid read data |

### Command Decoding

| `din[9:8]` | Operation |
|---|---|
| `00` | Store `din[7:0]` as internal write address |
| `01` | Write `din[7:0]` into `mem[wr_addr]` |
| `10` | Store `din[7:0]` as internal read address; de-assert `tx_valid` |
| `11` | Read `mem[rd_addr]` → `dout`; assert `tx_valid` |

### Parameters

| Parameter | Default | Description |
|---|---|---|
| `MEM_DEPTH` | 256 | Number of memory words |
| `ADDR_SIZE` | 8 | Address width (log₂ of MEM_DEPTH) |

## 6. FSM-Based Control

The SPI slave uses a **3-block FSM** architecture, following best practices taught in the course:

1. **State Memory** (`always @(posedge clk)`) — stores the current state; resets to IDLE.
2. **Next-State Logic** (`always @(*)`) — purely combinational; determines the next state based on `SS_n`, `MOSI`, `bit_count`, and `rd_addr_Done`.
3. **Output Logic** (`always @(posedge clk)`) — registered (Mealy-style outputs clocked); handles serial-to-parallel conversion, `rx_valid` assertion, MISO driving, and counter control.

### State Transition Diagram

```
                    ┌──────────┐
        SS_n = 1    │          │   SS_n = 1
     ┌─────────────→│   IDLE   │←──────────────┐
     │              │          │               │
     │              └────┬─────┘               │
     │                   │ SS_n = 0            │
     │              ┌────▼─────┐               │
     │              │          │               │
     │    ┌─────────│ CHK_CMD  │──────────┐    │
     │    │         │          │          │    │
     │    │         └────┬─────┘          │    │
     │    │              │                │    │
     │    │ MOSI=0       │ MOSI=1         │    │
     │    │              │ rd_addr_Done?   │    │
     │    │              │                │    │
     │  ┌─▼──────┐  ┌───▼──────┐  ┌──────▼──┐ │
     │  │        │  │          │  │         │ │
     ├──│ WRITE  │  │ READ_ADD │  │READ_DATA│─┘
     │  │        │  │          │  │         │
     │  └────────┘  └──────────┘  └─────────┘
     │       │            │             │
     └───────┴────────────┴─────────────┘
                  SS_n = 1
```

### States

| State | Encoding (Binary) | Description |
|---|---|---|
| `IDLE` | `3'b000` | Waiting for SS_n to go LOW |
| `CHK_CMD` | `3'b001` | First MOSI bit received; determines write vs. read |
| `WRITE` | `3'b010` | Shifting in 10-bit write command (address or data) |
| `READ_ADD` | `3'b011` | Shifting in 10-bit read-address command |
| `READ_DATA` | `3'b100` | Shifting in 10-bit read-data command, then shifting out MISO |

## 7. Reset Behaviour

Both the FSM and RAM use **synchronous active-low reset** (`rst_n`):

- **SPI_Slave on reset:**
  - `cs` → `IDLE`
  - `rx_data` → 0
  - `rx_valid` → 0
  - `MISO` → 0
  - `clr_counter` → 1 (counter is cleared)
  - `enable_counter` → 0

- **Single_Port_Sync_RAM on reset:**
  - `dout` → 0
  - `tx_valid` → 0
  - `wr_addr` → 0
  - `rd_addr` → 0

- **Up_Counter on reset:**
  - `count` → 0 (cleared by `clr` OR `!rst_n`)

## 9. Timing Comparison & Encoding Selection

The goal was to operate at the **highest possible frequency**.  The encoding with the **best (highest positive) setup and hold slack** after implementation was selected.

### Methodology

1. For each encoding, Vivado's **post-implementation timing report** was examined.
2. The **Worst Negative Slack (WNS)** for setup and **Worst Hold Slack (WHS)** were compared.
3. The encoding that yielded the **highest setup/hold slack** was chosen — indicating the most timing margin and therefore the highest achievable operating frequency.

### Result

**One-Hot encoding** was selected as the best encoding.  This is consistent with FPGA design best practices:

- FPGAs have abundant flip-flops but limited combinational logic.
- One-hot encoding uses one flip-flop per state, resulting in **simpler next-state logic** (fewer LUT levels → shorter critical paths → more timing slack).
- The folder name `One-Hot Encoding (The Best Encoding)` in the project confirms this selection.

The final bitstream for FPGA deployment was generated using the **one-hot encoding**.

## 12. Simulation & Verification

### Testbench (`SPI_Wrapper_tb.v`)

The testbench instantiates the top-level `SPI_Wrapper` and exercises all four SPI command types:

| Test | Description | Expected Result |
|---|---|---|
| **1 — Reset** | Assert `rst_n = 0`; check all memory = 0, all outputs = 0 | All zeros verified |
| **2 — Write Address** | Send command `00` + address `0x73` via MOSI | `wr_addr == 0x73` |
| **3 — Write Data** | Send command `01` + data `0x0D` via MOSI | `mem[0x73] == 0x0D` |
| **4 — Read Address** | Send command `10` + address `0x73` via MOSI | `rd_addr == 0x73` |
| **5 — Read Data** | Send command `11` + 8 dummy bits; observe MISO | Data `0x0D` shifted out on MISO |

### Memory Initialisation

The file `simulation/mem.dat` contains 256 lines of `00` (hex), loaded via `$readmemh`.  This ensures a clean, known starting state for all tests.

### Simulation Script (`RUN_SPI_Wrapper_TB.do`)

```tcl
vlib work
vlog ../src/SPI_Wrapper.v ../src/SPI_Slave.v ../src/Single_Port_Sync_RAM.v ../src/Up_Counter.v ../tb/SPI_Wrapper_tb.v
vsim -voptargs=+acc work.SPI_Wrapper_tb
add wave -position insertpoint  \
  sim:/SPI_Wrapper_tb/clk \
  sim:/SPI_Wrapper_tb/rst_n \
  sim:/SPI_Wrapper_tb/SS_n \
  ... (all internal signals) ...
run -all
```

The waveform displays:
- External signals: `clk`, `rst_n`, `SS_n`, `MOSI`, `MISO`
- FSM internals: `cs`, `ns`, `bit_count`, `rx_data_reg`, `rx_data`, `rx_valid`, `rd_addr_Done`
- RAM internals: `wr_addr`, `rd_addr`, `dout`, `tx_valid`, `tx_data`, `mem`

## 16. Lessons Learned

1. **FSM encoding matters for timing:** One-hot encoding consistently produced better timing slack on FPGAs due to simpler next-state decode logic, validating the theoretical advantage for FPGA fabrics.

2. **3-block FSM structure is clean and maintainable:** Separating state memory, next-state logic, and output logic into distinct always blocks made the design easier to debug, lint, and verify.

3. **Serial-to-parallel conversion requires careful bit counting:** The up-counter proved essential for tracking exactly which bit position is being received or transmitted, and the `clr`/`enable` control required precise coordination with FSM state transitions.

4. **The `rd_addr_Done` flag is critical:** Without it, the FSM cannot distinguish between a first-time read (READ_ADD) and a subsequent read (READ_DATA), since both start with MOSI = 1.

5. **QuestaLint provides confidence:** Achieving 100% design quality score (0 errors, 0 warnings) before synthesis catches potential issues early in the design flow.

6. **Debug cores (ILA) are invaluable for hardware verification:** Adding probes on MISO, MOSI, SS_n, rst_n, and clk allowed real-time observation of the SPI protocol on actual FPGA hardware.

7. **Constraint files do more than pin mapping:** The XDC file also defines the clock period, configuration voltage, bitstream compression, boot mode, and the entire ILA debug infrastructure.

8. **Simulation do-files must be correct and portable:** The do-file uses relative paths (`../src/`, `../tb/`) so it works regardless of the absolute installation directory.

## 17. Project Summary

This project implements a **fully functional SPI Slave interface** connected to a **256×8-bit Single-Port Synchronous RAM**, controlled by a **5-state FSM** written in Verilog. The design was verified through simulation in QuestaSim, statically analysed with QuestaLint (achieving 100% quality), synthesised and implemented in Vivado for three different state encodings (Binary, Gray, One-Hot), and prototyped on a Digilent Basys 3 FPGA. **One-Hot encoding** was selected as the optimal choice based on post-implementation timing analysis, offering the highest setup and hold slack. An ILA debug core was integrated to enable real-time hardware signal analysis, and a clean bitstream was generated for deployment.
